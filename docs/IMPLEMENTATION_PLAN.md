## 一、产品功能对标分析

### 1.1 核心功能对比矩阵

| 功能维度 | Spice (v0.3.2) | Octopus (v0.1.0) | 差距评估 | 优先级 |
|---------|----------------|------------------|---------|--------|
| 决策引擎 | 完整的决策策略、候选决策、决策追踪 | 基础决策引擎 + 多标准评分 | 中 | P0 |
| 协议标准 | SDEP v0.1 (完整规范 + JSON Schema) | ODEP (基础定义) | 大 | P0 |
| 工作区感知 | 只读感知：repo_map、search、git_status、read_file | LLM驱动意图提取 | 大 | P0 |
| 交互式CLI | 完整TUI (prompt_toolkit) + Rich渲染 | 简单命令行解析 | 大 | P0 |
| 会话管理 | 完整会话系统 (创建/切换/归档/搜索/时间线) | 无会话概念 | 大 | P1 |
| 批准机制 | approval-gated 显式批准检查点 | 无批准机制 | 大 | P1 |
| 执行器集成 | Codex、Claude Code、Hermes、OpenClaw、SDEP | 通用ToolRegistry | 大 | P1 |
| 记忆系统 | MemoryProvider + ContextCompiler | 多类型记忆 (情景/语义/程序/工作) | 小 | P2 |
| 伦理框架 | SafetyConstraint 集成在策略中 | 独立伦理框架 (十大原则) | ✅优势 | P2 |
| 模拟引擎 | 候选方案比较 | 蒙特卡洛模拟 | ✅优势 | P2 |
| 世界模型 | WorldState + WorldDelta | 实体-关系模型 | 中 | P2 |
| LLM支持 | OpenRouter、OpenAI、Anthropic、DeepSeek、MiMo | OpenAI + Ollama兼容 | 中 | P2 |
| 配置管理 | .spice/config.json + 配置向导 | 简单config模块 | 中 | P2 |

### 1.2 功能差距清单

**P0 - 必须补齐（核心竞争力）：**
1. 协议标准化（ODEP升级对标SDEP）
2. 工作区只读感知模块
3. 交互式CLI/TUI
4. 决策卡片渲染

**P1 - 重要补齐（生态能力）：**
5. 会话管理系统
6. 批准机制
7. 多执行器集成
8. 决策比较和细化

**P2 - 按需补齐（完善度）：**
9. 感知轮询模式
10. Doctor诊断
11. Quickstart脚手架
12. 多LLM提供商支持

### 2.2 核心模块划分

| 模块 | 职责 | 关键类/函数 | 依赖 |
|------|------|------------|------|
| runtime | 会话管理、状态持久化、上下文编译 | SessionStore, StateManager, ContextCompiler | LocalJsonStore |
| perception | 工作区感知、Git操作、URL读取 | WorkspacePerceiver, GitPerceiver, URLPerceiver | gitpython, requests |
| decision | 决策策略、候选决策、决策卡片 | DecisionPolicy, CandidateDecision, DecisionCardRenderer | rich, prompt_toolkit |
| execution | 工具执行、多执行器适配 | ExecutionLayer, ToolRegistry, ExecutorAdapter | ODEPProtocol |
| protocol | ODEP协议、消息序列化 | ODEPProtocol, MessageType, JSON Schema | jsonschema |
| memory | 多类型记忆、语义搜索 | LongTermMemory, MemoryIndex | numpy |
| ethics | 伦理审查、评分引擎 | EthicsFramework, EthicalRuleEngine | - |
| simulation | 蒙特卡洛模拟、情景推演 | SimulationEngine, ScenarioGenerator | - |
| llm | 多提供商LLM客户端 | LLMClient, OpenRouterProvider, AnthropicProvider | openai, anthropic-sdk |
| cli | 交互式命令行、TUI界面 | OctopusCLI, TUIShell, CommandHandler | argparse, rich, prompt_toolkit |

### 2.3 技术栈选型

| 维度 | 版本 | 说明 |
|------|------|------|
| Python | ≥3.11 | 更好的类型支持、性能优化 |
| CLI框架 | argparse + rich + prompt_toolkit | 对标Spice的TUI体验 |
| LLM支持 | openai + anthropic-sdk + openrouter | 多提供商支持 |
| Git操作 | gitpython | 工作区感知核心依赖 |
| HTTP请求 | requests | 已有依赖 |
| JSON Schema | jsonschema | 协议标准化必需 |
| 向量搜索 | numpy (轻量) / faiss (可选) | 记忆系统语义搜索 |

## 三、核心模块改造计划

### 3.1 第一阶段：核心基础能力（4周）

#### 3.1.1 ODEP v1.0协议
- 参考Spice SDEP设计完整协议规范
- 添加JSON Schema定义
- 支持多种传输方式

#### 3.1.2 工作区感知模块
- 只读工作区感知：repo_map、search、read_file、git_status、git_diff、git_log
- URL感知：网页文本读取
- 预算控制：最大文件数、字符数限制

#### 3.1.3 交互式CLI/TUI
- 使用prompt_toolkit实现交互式shell
- 使用rich渲染决策卡片和状态信息
- 支持多命令模式

#### 3.1.4 决策卡片渲染
- 实现决策卡片生成器
- 支持文本/JSON/Rich三种输出格式

### 3.2 第二阶段：会话与批准（3周）

- 会话管理系统
- 批准机制
- 决策比较功能

### 3.3 第三阶段：多执行器集成（4周）

- Executor接口定义
- dry_run、codex、claude_code、hermes、openclaw执行器
- SDEP兼容协议

### 3.4 第四阶段：完善与优化（3周）

- 感知轮询模式
- Doctor诊断工具
- Quickstart脚手架
- 多LLM提供商支持
- 性能优化与测试

## 五、质量保障方案

### 5.1 测试策略

- **单元测试**: 覆盖所有核心模块的基础功能，重点测试协议序列化/反序列化
- **集成测试**: 测试完整决策-执行流程，协议层与执行层交互测试
- **性能测试**: 决策响应时间测试，会话存储读写性能
- **测试覆盖率目标**: 单元测试 ≥80%，集成测试 ≥60%

### 5.2 验收标准

| 阶段 | 验收标准 |
|------|---------|
| v0.2 | CLI可运行、决策卡片输出、工作区感知可用 |
| v0.3 | 会话创建/切换/归档、批准流程可用 |
| v0.4 | 至少3种执行器可用、SDEP协议兼容 |
| v0.5 | 全部功能可用、性能达标、测试覆盖≥80% |
